# Docs AGENTS

本文件只补充 `docs/` 目录特有的规则；通用优先级、阅读顺序和最小变更原则以根目录 `AGENTS.md` 为准。

## 适用范围

- 本文件适用于 `docs/` 下除 `docs/spec/` 之外的基础库文档、模板和必要的设计记录。
- `docs/spec/` 是外部规范在本仓库中的只读引用源，当前仓库任务不得直接修改其中内容；如发现规范问题，应在交付说明中标注待反馈事项。
- 如果文档内容涉及具体 Go API、包行为、兼容性或测试策略，还需要阅读对应代码、README 和测试，不能只根据文档推断。
- 本文件只约束文档维护方式，不替代各模块 README、代码实现或 `docs/spec/` 正文中的具体规则。

## 目录放置

- 跨语言、跨实现、跨团队协作必须一致的规则只引用 `docs/spec/`，不要在当前仓库内直接修改或扩写。
- 语言或框架落地建议只引用 `docs/spec/guides/`；如果需要沉淀当前 Go 基础库的落地设计，应写入当前仓库自己的包文档或设计文档。
- `docs/` 下的目录可以和代码目录保持一致，例如 `docs/config/` 对应 `config/`，`docs/dkit/` 对应 `dkit/`。
- 单个 Go 包的快速使用说明优先放在该包目录的 `README.md`，例如 `config/README.md`；长期设计、兼容性说明或跨版本演进可以放在 `docs/<package>/`。
- 跨包主题可以放在 `docs/` 下独立英文目录中；如果主题主要服务某个包，优先收敛到 `docs/<package>/`。
- 新增文档前，先确认是否能补充到现有包 README、`docs/<package>/`、模板或设计文档，避免为一次局部迭代新建平行文档。
- 新建文档时，文件名和目录名使用英文；标题和正文语言跟随已有文档语境，当前默认使用中文。
- 仓库内链接使用可跳转的 Markdown 相对链接，不写本机绝对路径或 `file:///` 链接。

## 文档类型边界

- `spec`：只读引用源，定义互操作、兼容性或治理所必需的稳定规则，例如协议字段、header 名、错误类型、状态机、加载顺序、兼容边界。
- `guide`：只读引用源，说明某语言、框架或基础库组合的推荐落地方式；推荐项不能反向改变 spec 的兼容性定义。
- `README`：说明当前包或目录的定位、快速使用、行为边界、测试方式和常见注意事项。
- `template`：提供写作结构和检查项，不承载某个模块的现行设计结论。
- `ADR / design`：记录重要技术取舍、兼容演进、破坏性变更候选、跨包协作机制或长期维护边界；轻量变更不强制新增设计文档。
- 新增或重构技术设计文档时，优先使用 `technical-design-template.md` 作为模板；可按目标模块裁剪章节，但不应绕过其中对边界、兼容性、核心流程和验证方式的要求。

## 本地规则

- `docs/` 只沉淀基础库长期有效的技术知识，不写产品旅程、业务需求、融资计划、会议纪要、日常状态同步或执行流水。
- 文档应围绕库能力、规范约束、API 语义、兼容性、错误处理、并发与资源生命周期、可观测性和测试策略展开。
- 如果代码、测试或包 README 已经能清楚表达局部实现，不要在 `docs/` 重复维护实现清单；只保留稳定语义、边界、取舍和跨模块关系。
- 可读性优先于形式完整。先让读者快速看懂“模块解决什么问题、对外承诺什么、主链路如何工作”，再补充细节；不要为了覆盖模板把每个章节都写满。
- 简单 Bug 修复、局部示例修正或注释更新通常不需要新增设计文档；如果进入文档流程，应说明影响范围、兼容性判断和验证方式。
- 不把仅对当前 Go 实现成立的细节写成跨语言规范；这类内容应放在包 README、`docs/<package>/` 或实现设计中。
- 没有代码或测试依据的判断必须标注为待确认，不能写成已实现或强制规则。

## 技术设计内容边界

- 技术设计主要回答基础库模块“解决什么工程问题、对外提供什么稳定语义、内部如何保证这些语义、如何兼容演进”。
- 默认属于核心设计、应写入技术设计主体的内容包括：
  - 模块职责边界、非目标和与其他包的协作关系
  - 对外 API 的稳定语义、错误语义、零值语义和幂等性约束
  - 配置加载、上下文传递、重试、锁、ID 生成等关键流程的状态变化和失败路径
  - 并发安全、资源生命周期、超时、取消、重试、降级和清理策略
  - 兼容性边界、破坏性变更判断、迁移策略和废弃策略
  - 可观测性、测试矩阵、契约测试或跨实现一致性要求
- 默认更适合放到 README、guide 或代码注释，而不是技术设计主体的内容包括：
  - 完整 API 参数清单或逐项函数说明
  - 与某个示例应用绑定的业务字段、页面流程或部署流水
  - 临时兼容适配、一次性迁移脚本、过程记录和任务拆分
  - 代码已经直接表达且没有长期设计价值的局部实现细节
- 判断标准优先看“这段内容是否决定基础库长期如何工作、如何被依赖、如何兼容演进”，而不是只看它是否容易变化。
- 技术设计不是审计清单。只有真的影响依赖方理解或长期维护的内容，才值得进入主体；能一句话说清的内容，不要扩成整段或整表。

## 编辑规则

- 修改前先统一术语和文档边界，再调整字段表、示例和流程说明；不要留下前后两套口径。
- 引用 spec 或 guide 时优先链接已有规则，不重复定义同一概念；修改 README 或设计文档时应明确其推荐性质和适用前提。
- 表述模块关系、关键流程或跨包协作时，优先使用图或表格；如果文字和图同时存在，两者必须保持一致。
- 流程图和模块图里的关键连线应标明调用、读取、写入、派生、包装、传播、回调、清理或错误返回等语义，不只画无说明箭头。
- 同一张图里的抽象层级应一致；跨包关系图不要混入函数内部局部变量，函数内部机制图不要混入仓库级模块视角。
- 当一段规则已经用表格表达时，补充约束优先并入同一张表，避免字段定义、枚举解释和边界条件散落。
- 表述对象、字段、header、错误码或状态时，应明确名称、类型、必填性、兼容性和复用来源；不要只写抽象类型名。
- 示例必须可运行或明确标注为伪代码；示例中的包名、导入路径、环境变量和命令应与当前仓库一致。
- 修改命名、状态、枚举或术语后，要检索旧名称是否仍在相关文档中残留。
- 如果保留旧方案对比，必须明确它是背景或被废弃方案，不要让旧术语继续出现在现行规则描述里。
- 章节按需裁剪。迁移检查、完整测试矩阵、模块拆分取舍等内容，只有在本次设计确实依赖这些信息时才展开，不要求机械补齐。

## 迁移检查

- 当从其他项目迁入文档或重构旧文档时，必须逐章节判断内容去向，不能按原目录或原大类整体搬运。
- 可以迁入当前仓库的内容应落到包 README、`docs/<package>/`、template 或 design 中的合适位置；需要依赖 spec 或 guide 的内容只做引用。
- 不应迁入当前仓库的内容包括产品背景、用户旅程、页面交互、业务字段、融资或过程管理材料。
- 如果旧章节包含模块核心机制、兼容边界、错误语义或测试策略，即使它不是字段表或接口表，也不能因重构被省略。
- 如果某段旧内容不再迁移，应能说明原因：与当前基础库无关、与现代码不一致、已被新章节吸收，或属于过程/产品材料。

## 自查与交付

- 检查同一概念是否在 spec、guide、README、`docs/<package>/` 和模板之间出现强度冲突或重复定义。
- 检查表格、示例、流程图和正文描述是否同步更新。
- 检查所有相对链接、代码路径、包名、环境变量和命令是否有效。
- 检查是否把 Go 实现细节误写成跨语言强制规范，或把推荐写法误写成必须。
- 修改完成后至少做一次文档级诊断或格式自查。
- 交付时说明本次改了哪些文档、统一了哪些术语或结构，以及是否存在待确认问题。
