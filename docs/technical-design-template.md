# 技术设计模板

本文档用于基础库、规范实现、跨包协作机制或兼容性演进的技术设计，目标是澄清模块为什么这样设计、对外承诺什么稳定语义、内部如何保证这些语义，以及后续如何验证和演进。

写作时优先保证读者能顺着主线读懂，不要求把每个章节都写满。

使用原则：

- 技术设计应直接引用相关 spec、guide、README、代码和测试，不重复搬运完整使用说明；`docs/spec/` 只作为只读规范来源引用
- 文档核心在于说明库能力边界、API 语义、兼容性、失败路径和验证方式，而不是复述任务背景
- 优先补充到已有包 README、`docs/<package>/` 或现有设计文档；只有已有文档不适合承载时，才新建独立设计文档
- 只有当变更涉及跨包协作、公共 API 语义、兼容性、关键并发/资源机制或长期维护边界时，才使用这份模板
- 函数级参数清单、局部实现细节、一次性迁移脚本和示例应用代码不应挤占技术设计主体
- 输出技术设计前，必须阅读相关代码、测试和现有文档；没有代码依据或尚未确认的判断应标注为待确认
- 如果设计发现 `docs/spec/` 或 `docs/spec/guides/` 需要调整，只能在设计中引用并标注待反馈问题，不能在当前仓库任务中直接修改

迁移旧文档时额外遵守：

- 不能按原产品目录或“功能层 / 技术层 / API 层”粗拆，必须逐章节判断旧内容在基础库仓库中的新落点
- 旧文档里的核心机制、兼容边界、错误语义、测试策略或跨包协作逻辑不能因为不是字段表或接口表而省略
- 产品背景、用户旅程、页面状态、业务字段、会议过程和任务流水默认不迁入；如保留，应明确只是历史背景或外部追溯入口
- 如果某段旧内容不再迁移，应能明确说明是与当前基础库无关、与现代码不一致、已被新章节吸收，还是属于过程/产品材料

## 1. 背景与目标

先引用相关规范、README、代码或 Issue，再说明本次技术设计真正要解决的工程问题。

建议包含：

- 关联文档与代码：
- 当前模块或规范背景摘要：
- 本次技术目标：
- 非目标：
- 待确认问题：

建议写法：

- 先给一句话结论，说明“这个模块提供什么能力”
- 再用 3 到 5 个要点说明目标、非目标和约束
- 背景只保留帮助理解当前设计所必需的信息，不展开过程材料

## 2. 方案设计

这里聚焦核心方案本身，不单独维护泛化的“上下文与约束”章节；如有关键约束，直接放到对应 API、流程、模块或兼容性设计中说明。

### 2.1 模块边界与核心概念

- 核心概念：

- 模块职责：
- 非职责边界：
- 对象关系：优先使用图表达，并把关键连线语义直接标在图里；不要只画箭头，至少要让读者看出每条连线是在表达调用、包装、读取、写入、传播、派生、回调、关闭还是错误返回；只有关系过于复杂时才在图后补充说明
- 核心状态或资源：
- 核心概念和核心对象必须明确是否属于当前基础库模块；如果属于，应说明它是公共 API 对象、内部状态、配置对象、错误类型、资源句柄还是适配器抽象；如果不属于，应说明它是调用方对象、外部系统能力、规范概念或其他模块能力的聚合视图

推荐先写“职责 / 非职责 / 主对象关系”，只有在对象较多或容易混淆时再补对象表。

如果需要描述 API 对象、配置项、错误类型、状态或资源句柄，建议优先使用表格，并至少包含：

- 名称：对象名称
- 角色或职责：至少说明该对象做什么
- 是否公开：是否在公共 API 中暴露该对象

可选补充列：

- 来源：来自 spec、guide、当前实现还是待确认设计
- 备注：只写会影响理解的限制，不写泛泛描述

### 2.2 能力语义与兼容边界

说明对外能力承诺的稳定语义，以及哪些行为不承诺稳定。

建议回答：

- 公开函数以及对象是什么：
- 零值是否可用：
- 输入校验与默认值：
- 返回值与错误语义：
- 幂等性、并发安全或顺序保证：
- 资源创建、持有和释放责任：
- 兼容性边界：
- 废弃、迁移或破坏性变更策略：

推荐把“兼容性边界”直接写成一个短列表：

- 稳定承诺：依赖方可以据此编写代码或做迁移判断的行为
- 不承诺：实现可调整、调用方不应依赖的行为
- 待确认：当前仍需 review、代码验证或规范反馈的点

如果涉及具体字段、header、错误码或状态，建议使用表格，并说明来源是只读 spec、只读 guide、当前 Go 实现还是待确认设计。

### 2.4 模块设计

说明为了支撑核心对象、API 语义和核心流程，准备抽象哪些包、接口、适配器或内部组件，以及它们如何协作。

- 模块 1：

- 模块职责：
- 模块协作关系：优先使用图表达，并体现包边界、接口边界和依赖方向
- 如有必要，补充模块拆分的关键取舍：

如果模块本身存在稳定的内部工作机制，不能只停留在“模块协作图 + 主流程”两层，还应补充模块核心设计逻辑，例如：

- 核心模块的职责边界与非职责边界
- 模块输入、输出与中间状态如何切分
- 关键判断由调用方、公共 API、内部实现、适配器还是外部依赖负责
- 并发控制、资源生命周期、超时取消、重试、降级、清理或结束条件
- 为何采用当前拆分方式，而不是其他可选方案

对于 `config loader / context propagation / http client / distributed primitive / resource handle / adapter` 这类基础库模块，建议继续显式回答：

- 是否存在固定加载顺序、传播规则、重试规则、锁语义、ID 生成规则或释放规则
- 是否存在必须保留的兼容字段、header、错误类型、环境变量或默认值
- 运行时状态与单次调用上下文是否切分；如果切分，边界在哪里
- 哪些内容属于核心设计，哪些下沉到 README、guide、代码注释或测试说明

如果这些问题本次并不重要，可以不写；只有它们会影响外部依赖、实现边界或长期演进时，才需要展开。

### 2.6 迁移与边界检查（按需）

当本文档来自旧文档重构、拆分或规范化迁移时，建议在写作和自查时至少回答：

- 旧文档中的哪些章节迁移到了当前技术设计
- 哪些章节迁移到了包 README、`docs/<package>/` 或其他设计文档
- 哪些章节只保留为对 `docs/spec/` 或 `docs/spec/guides/` 的引用
- 哪些章节被代码、测试或现有文档吸收
- 哪些章节被明确降级为历史背景，不再作为现行规则
- 哪些章节因产品/业务/过程属性不再迁入当前仓库

如果本文档不是迁移而来，这一节可以省略。

如果没有做这个判断，后续很容易把“不是字段表、也不是接口表、但其实是核心机制”的内容误删。

基础库技术设计中的模块图优先体现包边界、接口边界和长期协作关系；局部函数调用、临时变量、具体测试夹具默认不进入图，除非它们本身就是本次设计的核心。

## 3. 使用说明

1. 这份模板用于基础库核心技术设计，不用于约束 spec、guide、README 或 ADR 的全部格式；`docs/spec/` 只读引用，不在当前仓库任务中直接修改。
2. 技术文档应直接引用相关规范、代码和测试，必要时只做简短摘要，不重复写完整使用说明。
3. 如果代码已经足以表达真实状态，文档可以不重复维护低价值实现清单，但仍应保留稳定语义、边界、兼容性和关键模块设计逻辑。
4. 当已有包 README、`docs/<package>/` 或设计文档足够承载本次设计时，优先增量补充，不新建平行文档；`docs/` 下目录可以与代码目录保持一致。
5. 如果本次是旧文档迁移，交付时应明确说明哪些旧章节被迁移到了哪里，哪些内容被有意保留为历史背景或不再迁入。
6. 推荐最小骨架是“背景与目标 + 方案设计 + 测试与验证”；只有在需要时再补迁移、对比方案或更细的模块拆分说明。
